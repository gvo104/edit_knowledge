# NIR_KE — Knowledge Editing / Knowledge Probing Pipeline

## 1. Идея и смысл проекта

Проект решает практическую исследовательскую задачу: **как добавлять редкие факты в языковую модель** и при этом **минимально портить уже имеющиеся знания**.

Фокус проекта:
- построить воспроизводимый стенд для сравнения методов редактирования знаний;
- измерять качество **до** и **после** редактирования по единым метрикам;
- сравнивать методы на одинаковых данных, запросах и условиях запуска.

Ключевая предметная зона на текущем этапе — биомедицинские данные (PubTator), но архитектура сделана расширяемой под другие специализированные домены.

## 2. Как скачать и запустить

### 2.1. Клонирование

```bash
git clone <URL_ВАШЕГО_РЕПОЗИТОРИЯ>
cd NIR_KE/edit_knowledge
```

### 2.2. Окружение

В проекте используется conda-окружение (пример в `environment.yml`).

```bash
conda env create -f environment.yml
conda activate nir_env
```

Если окружение уже есть, убедитесь, что установлены минимум:
- Python 3.9+
- PyTorch
- transformers
- peft
- pyyaml

### 2.3. Один главный запуск полного пайплайна

```bash
python src/cli/main.py --config configs/experiment/full_pipeline.yaml
```

Это основной и рекомендуемый способ запуска.

---

## 3. Что делает полный запуск

Один запуск последовательно выполняет:
1. `prepare_data`
2. `baseline_probe`
3. `train_edit_method` (для каждого метода из конфига)
4. `post_edit_probe` (для каждого метода)
5. `aggregate_results`

Все артефакты каждого шага сохраняются отдельно и связаны через `manifest.json`.

## 5. Подробное описание пайплайна

### 5.1. `prepare_data`

Что происходит:
- чтение PubTator-данных;
- выборка статей;
- построение SPO-триплетов;
- генерация тестовых запросов:
  - direct
  - inverse
  - paraphrase
  - locality

Результат стадии:
- `artifacts/triplets.json`
- `manifest.json`
- `config.yaml`

### 5.2. `baseline_probe`

Что происходит:
- загрузка базовой модели;
- генерация ответов на тестовые запросы;
- оценка качества на baseline-модели.

Результат стадии:
- `artifacts/baseline_probing_raw.json`
- `metrics/metrics.json` (нормализованный формат)

### 5.3. `train_edit_method`

Что происходит:
- для каждого метода из списка в конфиге запускается редактирование/обучение;
- метод работает через единый интерфейс:
  - `prepare(...)`
  - `train_or_edit(...)`
  - `save(...)`
  - `load(...)`
  - `apply_to_model(...)`

Результат стадии:
- артефакты конкретного метода (например, LoRA-адаптер)
- `manifest.json` с путями входа/выхода

### 5.4. `post_edit_probe`

Что происходит:
- probing отредактированной модели;
- расчёт тех же метрик, что и для baseline.

Результат стадии:
- `artifacts/post_edit_probing_raw.json`
- `metrics/metrics.json`

### 5.5. `aggregate_results`

Что происходит:
- сбор baseline и post-edit метрик;
- сравнение методов между собой;
- baseline vs edited анализ;
- построение табличного представления.

Финальный отчёт:
- `outputs/reports/full_pipeline_v1_YYYYMMDD_HHMMSS_run_report.json`

## 7. Нормализованные метрики

Новый слой метрик: `src/probing/metrics/`.

Поддерживается единый формат для категорий:
- `direct`
- `inverse`
- `paraphrase`
- `locality`

Для каждой категории считаются:
- `total`
- `perfect_1_0`
- `accuracy_1_0`
- `good_enough_count`
- `good_enough_rate`
- `mean_score`

В финальном отчёте есть блоки для табличной аналитики:
- `rows` (dataset / method / metric_name / value)
- `summary_by_method`
- `summary_by_dataset`
- `best_method_per_dataset`

## 8. Текущий этап развития проекта

Проект находится в переходе от первоначального прототипа к исследовательской платформе сравнения методов.

### Уже сделано
- единый full-pipeline entrypoint;
- модульная архитектура `src/`;
- отдельный слой нормализованных метрик;
- структурированное хранение артефактов по стадиям;
- базовый метод LocFT-BF (LoRA SFT) + вариант с augmentation;
- интеграция ROME как заглушки через единый интерфейс.

### Ограничения текущего состояния
- ROME/MEMIT/MEND/WISE пока не полноценные реализации;
- некоторые исследовательские метрики (Edit Success, Reversal Score) ещё требуют окончательной формализации в текущем коде агрегатора;
- пока основная отладка выполнена на PubTator.

---

## 9. Что планируется доделать в ближайшее время

1. Полноценная интеграция ROME (не заглушка) в том же интерфейсе `editing`.
2. Подключение минимум одного метода из пары MEMIT/MEND.
3. Расширение метрик до полного набора:
   - Edit Success
   - Reversal Score
   - Paraphrase Robustness
   - Locality
4. Поддержка 1–2 дополнительных специализированных датасетов, включая русский (желательно геология).
5. Добавление второй модели для сравнения (например, Phi или LLaMA).
6. Финализация отчёта: автоматические сводные таблицы/графики forgetting curve.

## 10. Итоговый план исследования

### Этап 1. Определение рамок исследования и выбор методов

**Цель:** зафиксировать, какие методы реально успеть сравнить при ограниченных ресурсах.

**Задачи:**
- Оценить применимость методов:
  - LocFT-BF;
  - ROME;
  - MEMIT;
  - MEND;
  - WISE;
  - Model Merging — только как опциональный метод.
- Проверить доступность реализаций, сложность интеграции, требования к памяти и времени.
- Сформировать финальный набор методов для экспериментов.

**Финальный состав методов для работы:**
- Основной метод: LocFT-BF (LoRA SFT).
- Второй базовый метод: ROME.
- Дополнительный метод 1: LocFT-BF + аугментация.
- Дополнительный метод 2: один из EasyEdit-методов — MEMIT или MEND.
- WISE — только если подключение не затормозит проект.
- Model Merging — резервный вариант.

**Результат этапа:**
- утверждённый список методов;
- критерии сравнения;
- приоритет экспериментов.

### Этап 2. Подготовка данных, моделей и инфраструктуры

**Цель:** собрать воспроизводимую базу для probing и editing.

**Задачи:**
- Подобрать датасеты:
  - биомедицинский датасет из PubTator Central;
  - ещё 1–2 специализированных датасета;
  - желательно один на русском языке и по геологии;
  - при необходимости собрать собственный датасет из открытых документов.
- Подготовить триплеты из документов.
- Выбрать базовые модели:
  - Qwen3.5 small (0.8B–9B);
  - fallback: Qwen2.5;
  - дополнительная: LLaMA или Phi.
- Подготовить разбиение train/test без утечек.
- Сформировать тестовые запросы: прямые, обратные, парафразы, locality.
- Настроить формат знаний и метаданных для воспроизводимости.

**Результат этапа:**
- датасеты в едином формате;
- список моделей;
- стандартизированные тестовые запросы;
- готовая инфраструктура probing.

### Этап 3. Knowledge probing до редактирования

**Цель:** измерить исходное качество модели.

**Задачи:**
- Запустить baseline probing на всех датасетах.
- Измерить:
  - Edit Success;
  - Reversal Score;
  - Paraphrase Robustness;
  - Locality.
- Зафиксировать результаты в едином формате.
- Сохранить конфиги и manifests.

**Результат этапа:**
- baseline-оценки;
- единый отчёт до редактирования;
- воспроизводимые артефакты.

### Этап 4. Эксперименты по редактированию знаний

**Цель:** сравнить методы редактирования на одинаковых данных и метриках.

**Задачи:**
- Прогнать:
  - LocFT-BF (LoRA SFT);
  - LocFT-BF + аугментация;
  - ROME;
  - MEMIT / MEND / WISE (минимум 2 метода при доступных ресурсах).
- Для LocFT-BF:
  - обучение на QA-парах по редким фактам;
  - добавление парафраз и соседних фактов.
- Для ROME:
  - последовательные правки;
  - сохранение промежуточных состояний.
- Для EasyEdit-методов:
  - единый интерфейс запуска и сравнения.
- Дополнительно (при необходимости):
  - влияние числа примеров;
  - влияние аугментации;
  - влияние последовательности правок.

**Результат этапа:**
- минимум 3 версии отредактированной модели;
- промежуточные состояния;
- артефакты для анализа забывания.

### Этап 5. Оценка, сравнение и анализ

**Цель:** определить лучший метод для добавления редкой терминологии.

**Задачи:**
- Повторно прогнать probing после каждого метода.
- Сравнить с baseline.
- Построить forgetting curve.
- Оценить:
  - глубину усвоения;
  - устойчивость к переформулировкам;
  - влияние на общий уровень знаний;
  - катастрофическое забывание;
  - компромисс точность редактирования vs locality.
- Свести сравнение в единую таблицу.

**Результат этапа:**
- количественное сравнение методов;
- графики и таблицы;
- выводы по пригодности методов.

### Этап 6. Итоговый отчёт и рекомендации

**Цель:** оформить завершённую исследовательскую работу.

**Задачи:**
- Свести результаты всех экспериментов.
- Сформулировать рекомендации:
  - какой метод лучше для редких фактов;
  - когда применять локальное дообучение;
  - когда использовать точечное редактирование;
  - как снижать забывание;
  - насколько помогает аугментация.
- Описать методологию, данные, модели, метрики и ограничения.
- Подготовить финальный отчёт и документацию.

**Результат этапа:**
- завершённое исследование;
- аргументированные выводы;
- воспроизводимый пайплайн;
- готовая документация.
