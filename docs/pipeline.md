# Edit Knowledge Pipeline – документация (базовый скелет)


## 1. Общая идея

Мы строим измерительный стенд для оценки и последующего **редактирования фактических знаний** языковых моделей.  
На первом этапе реализован **knowledge probing** на биомедицинском датасете PubTator.

**Что уже работает:**
- Извлечение структурированных триплетов (ген / болезнь / мутация – PMID) из файлов `*2pubtatorcentral`.
- Автоматическая генерация тестовых запросов трёх типов (прямой, обратный, парафраз).
- Прогон модели (`Qwen2.5-0.5B-Instruct`) в режиме жадной генерации с пакетной обработкой (batching).
- Оценка ответов (улучшенная метрика, учитывающая PMID и гены).
- Версионирование выходных файлов и поддержка нескольких датасетов.

## 2. Структура проекта

Весь код лежит в `edit_knowledge/`. Его окружение – корень проекта `NIR_KE/`.  
Данные находятся в `edit_knowledge/data/`.

```
edit_knowledge/
├── config.py                      # все настройки, выбор датасета, версионирование
├── main.py                        # главная точка входа (запускает оба этапа)
├── data/                          # все данные
│   ├── PubTator/                  # сырые файлы (gene2pubtatorcentral, disease2... и т.д.)
│   ├── triplets/                  # результаты обработки (*_triplets.json)
│   └── probing_results/           # результаты probing (*_probing_results.json)
├── data_processing/               # этап 1: от PubTator до триплетов
│   ├── run_processing.py          # скрипт запуска обработки (с аргументами)
│   ├── data_reader.py             # потоковое чтение (поддержка .gz)
│   ├── stats.py                   # сбор статистики без загрузки в память
│   ├── triplet_extractor.py       # группировка сущностей, построение SPO-триплетов
│   └── query_generator.py         # генерация тестовых вопросов (direct/inverse/paraphrase)
├── llm/                           # общие модули для работы с LLM (переиспользуемые)
│   ├── model_loader.py            # загрузка модели + токенизатора
│   └── query_engine.py            # генерация ответов (одиночная + пакетная) и метрики оценки
├── model_probing/                 # этап 2: probing модели
│   ├── run_probing.py             # скрипт запуска probing (с аргументами)
│   ├── probing_runner.py          # логика пакетного получения ответов и оценки
│   └── results_handler.py         # вычисление сводной статистики и сохранение
└── docs/
    └── pipeline.md                # этот документ
```

## 3. Конфигурация (`config.py`)

Все параметры вынесены в `config.py`. Его редактируют вручную, либо переопределяют аргументами командной строки (где это предусмотрено).

**Ключевые секции:**

### Выбор датасета
- `DATASET_NAME` – имя по умолчанию (`"gene"`).
- `DATASET_CONFIGS` – словарь с путями и типами сущностей для gene, disease, mutation.
- Функция `get_dataset_config(name)` возвращает нужный конфиг.

### Обработка данных
- `SAMPLE_SIZE = 50` – сколько уникальных PMID брать для построения триплетов.
- `MAX_COOCCUR_GENES = 5` – ограничение числа пар ко-оккуренции внутри статьи.

### Версионирование
- `VERSION_TIMESTAMP` – метка времени, включается в имена выходных файлов.

### Probing
- `MODEL_NAME = "Qwen/Qwen2.5-0.5B-Instruct"`.
- `BATCH_SIZE = 16`, `MAX_NEW_TOKENS = 30`, `EVALUATION_STRATEGY = "improved"`.

### Сохранение
- `OUTPUT_DIR`, `PROBING_DIR` – каталоги для результатов.
- Шаблоны имён файлов с плейсхолдерами `{dataset}`, `{version}`.

## 4. Этап 1 – обработка данных

**Назначение:** превратить сырой табулированный файл PubTator в структурированные SPO‑триплеты и тестовые запросы.

**Запуск:**
```bash
python edit_knowledge/data_processing/run_processing.py --dataset gene
```
По умолчанию берётся `gene`. Можно указать `disease` или `mutation`.

**Алгоритм:**
1. **Статистика** – потоковый проход по всему файлу (без загрузки в память) собирает число записей, уникальных PMID и уникальных сущностей.
2. **Выборка** – второй проход, отбирающий первые `SAMPLE_SIZE` уникальных PMID (вместе со всеми их аннотациями).
3. **Группировка** – сущности объединяются по PMID: `{pmid: [entity1, entity2, ...]}`.
4. **Триплеты** – из каждой статьи создаются два типа фактов:
   - `(entity, "mentioned in", "PMID:...")` – упоминание в статье.
   - `(entityA, "co-occurs with", entityB)` – ко‑оккуренция (первые `MAX_COOCCUR_GENES` сущностей).
5. **Тестовые запросы** – на основе шаблонов для типа сущности (gene/disease/mutation) генерируются:
   - **direct** – прямой вопрос («В какой статье упоминается X?»).
   - **inverse** – обратный вопрос («Какая сущность упоминается в статье Y?»).
   - **paraphrase** – перефразированный вопрос.
6. **Сохранение** – результат записывается в `data/triplets/{dataset}_{version}_triplets.json`.

**Особенности:**
- Поддержка сжатых файлов (`.gz`) прозрачна.
- Поля сущностей (`entity_id`, `entity_name`) унифицированы для всех типов датасетов.
- Версионирование предотвращает перезапись результатов.

## 5. Этап 2 – Knowledge Probing

**Назначение:** измерить, насколько хорошо модель помнит конкретные биомедицинские факты.

**Запуск:**
```bash
python edit_knowledge/model_probing/run_probing.py --dataset gene
```
Можно указать конкретную версию триплетов через `--version`, иначе возьмётся последняя.

**Алгоритм:**
1. **Загрузка триплетов** – из файла `data/triplets/{dataset}_*_triplets.json`.
2. **Загрузка модели** – `Qwen2.5-0.5B-Instruct` в `float16` на GPU. Если GPU нет, поставить `DEVICE = "cpu"` в конфиге.
3. **Генерация ответов** – все вопросы обрабатываются **пакетно** (batch) для скорости.
   - Жадная генерация (`do_sample=False`) – детерминированно, без случайности.
   - `max_new_tokens = 30` (PMID и короткие названия помещаются).
4. **Оценка** – используется **улучшенная метрика**:
   - 1.0 – точное совпадение строк.
   - 0.9 – ожидаемый ответ содержится в ответе модели.
   - 0.85 – совпадает PMID (регулярное выражение `PMID:?\s*(\d+)`).
   - 0.7 – совпадает название гена (упрощённое выделение).
   - 0.0 – неверно.
5. **Статистика** – по каждому типу запросов считаются:
   - `perfect_1.0` (полное совпадение) и `accuracy_1.0`.
   - `good_enough_count` (≥0.8) и доля.
   - `mean_score`.
6. **Сохранение** – результаты записываются в `data/probing_results/{model_short}_{dataset}_{version}_probing_results.json`.

**Оптимизации:**
- Пакетная обработка (`batch_size=16`) ускоряет инференс в 3–5 раз.
- Все предупреждения `transformers` (о неиспользуемых `temperature`, `top_p`) подавлены.
- Функция `get_model_answers_batch` корректно обрабатывает padding и обрезает входные токены.

## 6. Главный скрипт `main.py`

Запускает оба этапа одной командой:
```bash
python edit_knowledge/main.py --dataset gene
```
Флаги:
- `--dataset` (gene/disease/mutation)
- `--skip_processing` – пропустить обработку данных, если уже выполнена.
- `--version` – указать конкретную версию триплетов для probing.

-`python main.py --all_datasets --skip_processing`

**Принцип:** каждый этап вызывается через `subprocess` в том же окружении, чтобы сохранить изоляцию импортов.

## 7. Результаты baseline (gene, Qwen2.5-0.5B)

**Метрики:**
| Тип запроса | Кол-во | Perfect (1.0) | Good enough (≥0.8) | Mean Score |
|-------------|--------|---------------|---------------------|------------|
| direct      | 100    | 0 (0%)        | 5 (5%)              | 0.045     |
| inverse     | 72     | 0 (0%)        | 0 (0%)              | 0.000     |
| paraphrase  | 72     | 0 (0%)        | 0 (0%)              | 0.000      |

Модель **не знает** точных PMID и связей генов, склонна к галлюцинациям. Это ожидаемо для 0.5B параметров и является отправной точкой для экспериментов по редактированию знаний.

## 9. Важные замечания

- Окружение: Python 3.9, conda env `nir_env` с PyTorch, transformers >= 4.44.
- Все скрипты должны запускаться из корня `NIR_KE/`, либо из `edit_knowledge/` с корректным `sys.path`.
- Большие файлы PubTator обрабатываются поточно, памяти хватает.
- При смене модели необходимо проверить совместимость `chat_template` (для Qwen2.5 используется стандартный шаблон).
