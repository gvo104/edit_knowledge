# Edit Knowledge Pipeline – документация (v2.0)

**Версия:** 2025-05-23 (актуально для Python 3.9 + conda env `nir_env`)  
**Автор:** команда проекта NIR_KE

---

## 1. Общая идея

Мы строим измерительный стенд для оценки и последующего **редактирования фактических знаний** языковых моделей.  
На первом этапе реализован **knowledge probing** на биомедицинском датасете PubTator, а затем – первый метод редактирования **LocFT-BF (LoRA SFT)**.

**Что уже работает:**
- Извлечение структурированных триплетов (ген / болезнь / мутация – PMID) из файлов `*2pubtatorcentral`.
- Автоматическая генерация тестовых запросов:
  - прямые вопросы (direct)
  - обратные вопросы (inverse)
  - множественные парафразы (3–4 варианта)
  - несвязанные общие вопросы для проверки сохранения знаний (locality)
- Прогон модели (`Qwen2.5-0.5B-Instruct`) в режиме жадной генерации с пакетной обработкой (batching).
- Оценка ответов (улучшенная метрика, учитывающая PMID и названия сущностей).
- **Дообучение модели с помощью LoRA** (LocFT-BF) на основе сгенерированных триплетов.
- **Автоматический probing после обучения** и **сравнение метрик до/после**.
- Версионирование всех выходных файлов и поддержка нескольких датасетов (gene, disease, mutation).
- Единый запуск полного цикла через **матрицу экспериментов** (`--debug`).
- Сбор общего отчёта (`data/summary_*.json`) и вывод итоговой таблицы в консоль.

## 2. Структура проекта

Весь код лежит в `edit_knowledge/`. Его окружение – корень проекта `NIR_KE/`.  
Данные находятся в `edit_knowledge/data/`.

```
edit_knowledge/

├── data/                          # все данные

│   ├── lora_checkpoints/          # временные чекпойнты обучения
│   ├── probing_results/           # результаты baseline probing (*_probing_results.json)
│   ├── probing_results_lora/      # результаты probing после LoRA
│   └── summary_*.json             # сводный отчёт по всем датасетам
├── data_processing/               # этап 1: от PubTator до триплетов и тестовых запросов

│   ├── data_reader.py             # потоковое чтение (поддержка .gz)

│   ├── triplet_extractor.py       # группировка сущностей, построение SPO-триплетов

│   ├── model_loader.py            # загрузка модели + токенизатора
```

## 3. Конфигурация (`config.py`)

Все параметры вынесены в `config.py`. Его можно редактировать вручную, либо переопределять аргументами командной строки (где это предусмотрено).

**Ключевые секции:**

### Выбор датасета
- `DATASET_NAME` – имя по умолчанию (`"gene"`).
- `DATASET_CONFIGS` – словарь с путями и типами сущностей для gene, disease, mutation.
- Функция `get_dataset_config(name)` возвращает нужный конфиг.

### Обработка данных
- `SAMPLE_SIZE = 50` – сколько уникальных PMID брать для построения триплетов.
- `MAX_COOCCUR_GENES = 5` – ограничение числа пар ко-оккуренции внутри статьи.

### Версионирование
- `VERSION_TIMESTAMP` – метка времени, включается в имена выходных файлов.

### Probing
- `MODEL_NAME = "Qwen/Qwen2.5-0.5B-Instruct"`.
- `BATCH_SIZE = 16`, `MAX_NEW_TOKENS = 30`, `EVALUATION_STRATEGY = "improved"`.
- `GOOD_ENOUGH_THRESHOLD = 0.8` – порог для подсчёта «хороших» ответов.

### Сохранение
- `OUTPUT_DIR`, `PROBING_DIR`, `PROBING_DIR_LORA` – каталоги для результатов.

## 4. Этап 1 – обработка данных

**Назначение:** превратить сырой табулированный файл PubTator в структурированные SPO‑триплеты и тестовые запросы.

**Запуск:**
```bash
python edit_knowledge/data_processing/run_processing.py --dataset gene
```
По умолчанию берётся `gene`. Можно указать `disease` или `mutation`.

**Алгоритм:**
1. **Статистика** – потоковый проход по всему файлу (без загрузки в память) собирает число записей, уникальных PMID и уникальных сущностей.
2. **Выборка** – второй проход, отбирающий первые `SAMPLE_SIZE` уникальных PMID (вместе со всеми их аннотациями).
3. **Группировка** – сущности объединяются по PMID: `{pmid: [entity1, entity2, ...]}`.
4. **Триплеты** – из каждой статьи создаются два типа фактов:
   - `(entity, "mentioned in", "PMID:...")` – упоминание в статье.
   - `(entityA, "co-occurs with", entityB)` – ко‑оккуренция (первые `MAX_COOCCUR_GENES` сущностей).
5. **Тестовые запросы** – на основе шаблонов для типа сущности (gene/disease/mutation) генерируются:
   - **direct** – прямые вопросы («В какой статье упоминается X?»)
   - **inverse** – обратные вопросы («Какая сущность упоминается в статье Y?»)
   - **paraphrase** – 3–4 перефразированных варианта для каждого факта
   - **locality** – 10 общеизвестных вопросов (столицы, история, наука) для проверки сохранения общих знаний
6. **Сохранение** – результат записывается в `data/triplets/{dataset}_{version}_triplets.json`. Файл содержит триплеты, все тестовые запросы (включая locality) и метаданные.

**Особенности:**
- Поддержка сжатых файлов (`.gz`) прозрачна.
- Поля сущностей (`entity_id`, `entity_name`) унифицированы для всех типов датасетов.
- Версионирование предотвращает перезапись результатов.

## 5. Этап 2 – Knowledge Probing

**Назначение:** количественно оценить, насколько модель знает биомедицинские факты и сохраняет общие знания.

**Запуск:**
```bash
python edit_knowledge/model_probing/run_probing.py --dataset gene
```
Можно указать конкретную версию триплетов через `--version`; иначе возьмётся последняя по алфавиту (самая свежая временная метка).
При указании `--lora_path` модель загружается с дообученным адаптером, а результаты сохраняются в отдельную папку `probing_results_lora`.

**Алгоритм:**
1. **Загрузка триплетов и запросов** – из файла `data/triplets/{dataset}_*_triplets.json`.  
   Тестовые запросы включают четыре категории: `direct`, `inverse`, `paraphrase`, `locality`.
2. **Загрузка модели** – `Qwen2.5-0.5B-Instruct` в `float16` на GPU. Если GPU нет, можно выставить `DEVICE = "cpu"` в конфиге.
3. **Генерация ответов** – все вопросы обрабатываются **пакетно** (batch) для скорости.
   - Жадная генерация (`do_sample=False`) – детерминированно, без случайности.
   - `max_new_tokens = 30` (PMID и короткие названия помещаются).
4. **Оценка** – используется **улучшенная метрика**:
   - 1.0 – точное совпадение строк.
   - 0.9 – ожидаемый ответ содержится в ответе модели.
   - 0.85 – совпадает PMID (регулярное выражение `PMID:?\s*(\d+)`).
   - 0.7 – совпадает название сущности (упрощённый поиск).
   - 0.0 – неверно.
5. **Статистика** – по каждой категории запросов считаются:
   - `total`, `perfect_1.0`, `accuracy_1.0`
   - `good_enough_count` (≥0.8) и `good_enough_rate`
   - `mean_score`
6. **Сохранение** – результаты записываются в `data/probing_results/` (baseline) или `data/probing_results_lora/` (после LoRA).  
   При наличии `--lora_path` автоматически выполняется сравнение с последним baseline и выводятся изменения метрик.

**Оптимизации:**
- Пакетная обработка (`batch_size=16`) ускоряет инференс в 3–5 раз.
- Все предупреждения `transformers` о неиспользуемых параметрах сэмплирования подавлены.
- Функция `get_model_answers_batch` корректно обрабатывает padding и обрезает входные токены.

## 6. Этап 3 – LocFT-BF (SFT + LoRA)

**Назначение:** дообучить модель на фактах из триплетов с помощью LoRA, чтобы она запомнила редкие биомедицинские знания.

**Запуск:**
```bash
python edit_knowledge/model_training/run_sft_lora.py --dataset gene
```
Принимает `--version` для указания конкретной версии триплетов.  
По умолчанию после обучения автоматически запускает probing с полученным адаптером (можно отключить флагом `--no_probe`).

**Алгоритм:**
1. **Подготовка SFT-датасета** (`prepare_sft_dataset.py`) – из триплетов извлекаются все direct, inverse и paraphrase запросы, формируются QA-пары в формате чата Qwen2.5. Сохраняется в `data/training/`.
2. **Загрузка модели** – та же `Qwen2.5-0.5B-Instruct`.
3. **Применение LoRA** – конфигурация: `r=8, lora_alpha=32, target_modules=["q_proj","v_proj"]`. Обучается всего 0.11% параметров.
4. **Обучение** – 5 эпох, learning rate 2e-4, batch size 4, без Trainer’а (простой цикл PyTorch). Чекпойнты не сохраняются (только финальный адаптер).
5. **Сохранение адаптера** – в `data/lora_adapters/{dataset}_{version}/`.
6. **Автоматический probing** – если не отключён, запускает `run_probing.py` с `--lora_path`, что даёт метрики после дообучения и сравнение с baseline.

**Особенности:**
- Не требует библиотеки `datasets` (во избежание конфликтов версий).
- Проверяет существование адаптера – если уже есть, повторно не обучает (можно переопределить флагом `--force_retrain`).
- Встроен в матрицу `main.py --debug`.

## 7. Главный скрипт `main.py`

### Обычный режим
```bash
python edit_knowledge/main.py --dataset gene
```
Флаги:
- `--dataset` (gene/disease/mutation)
- `--all_datasets` – запустить обработку и/или probing для всех датасетов
- `--skip_processing` – пропустить обработку данных
- `--skip_probing` – пропустить probing
- `--version` – указать конкретную версию триплетов

### Режим матрицы экспериментов (`--debug`)
Позволяет гибко задать, какие этапы выполнять для каких датасетов.  
Матрица `DEBUG_MATRIX` описана прямо в `main.py`. Пример:
```python
DEBUG_MATRIX = {
    "gene": {"processing": True, "probing": True, "train_lora": True, "probe_after_train": True},
    "disease": {"processing": True, "probing": True, "train_lora": True, "probe_after_train": True},
    ...
}
```
После завершения всех экспериментов автоматически:
- Собирается сводка последних baseline и LoRA метрик в `data/summary_YYYYMMDD_HHMMSS.json`.
- Выводится аккуратная таблица сравнения в консоль.

**Запуск полного цикла с пересозданием всех данных:**
```bash
python main.py --debug
```

## 8. Результаты (после внедрения LoRA)

Пример сводной таблицы (gene, disease, mutation – все три датасета):

| Категория   | Метрика         | Baseline | LoRA   | Δ      |
|-------------|-----------------|----------|--------|--------|
| direct      | mean_score      | 0.0180   | 0.0840 | +0.066 |
|             | perfect (1.0)   | 0.0000   | 0.0500 | +0.050 |
|             | good (≥0.8)     | 0.0200   | 0.0800 | +0.060 |
| inverse     | mean_score      | 0.0000   | 0.0264 | +0.026 |
| paraphrase  | mean_score      | 0.0105   | 0.0977 | +0.087 |
| locality    | mean_score      | 0.7200   | 0.0000 | –0.720 |

**Наблюдения:**
- LoRA значительно улучшает знание целевых фактов (direct + paraphrase).
- Обратные вопросы всё ещё даются тяжело (inverse почти не растёт).
- **Критическая проблема:** общие знания (locality) полностью теряются после дообучения — mean score падает с 0.72 до 0.0. Это указывает на катастрофическое забывание и требует доработки методики (регуляризация, меньший learning rate, смешивание с общими данными).

## 9. Дальнейшие шаги (планы)

- **Улучшить LocFT-BF:**
  - добавить в обучающую выборку общеизвестные вопросы (locality-запросы) или перемешать их с целевыми фактами для борьбы с катастрофическим забыванием;
  - экспериментировать с рангом LoRA, количеством эпох, learning rate и объёмом фактов (SAMPLE_SIZE).
- **Реализовать LocFT-BF с аугментацией (KEDAS):**
  - расширить обучающие QA-пары дополнительными парафразами, синонимичными формулировками и соседними фактами (согласно статье KEDAS).
- **Подготовить дополнительные датасеты:**
  - найти ещё 1–2 специфичных датасета, желательно один на русском языке (например, геология);
  - при необходимости собрать собственный датасет из открытых источников и извлечь из него триплеты.
- **Расширить probing:**
  - увеличить количество общеизвестных вопросов для более точной оценки Locality;
  - сформировать тестовые запросы для новых датасетов.
- **Внедрить ROME (Rank-One Model Editing):**
  - использовать реализацию из EasyEdit (отдельное окружение) для точечного редактирования;
  - провести последовательное внесение правок с сохранением промежуточных состояний модели;
  - сравнить с LocFT-BF по метрикам Edit Success, Reversal Score, Paraphrase Robustness, Locality.
- **Протестировать MEMIT / WISE:**
  - MEMIT – массовое редактирование, устойчивость к последовательным правкам;
  - WISE – применить имеющуюся оптимизацию (с привлечением разработчика);
  - сравнить с ROME и LoRA по глубине усвоения знаний и влиянию на общее качество модели.

## 10. Важные замечания

- **Окружение:** Python 3.9, conda env `nir_env` с PyTorch, transformers >= 4.44, peft.  
  *Примечание:* EasyEdit требует transformers 4.30.1, поэтому для ROME/MEMIT потребуется отдельное окружение.
- Все скрипты запускаются из корня `NIR_KE/` или из `edit_knowledge/` с корректным `sys.path`.
- Большие файлы PubTator обрабатываются поточно, не загружая весь объём в память.
- Версионирование гарантирует, что каждый запуск создаёт новый файл, не затирая предыдущие результаты.
- При использовании LoRA в probing результаты автоматически сохраняются в отдельную папку `probing_results_lora`.
